Founding Manifesto

Why we build
the way we build

Bloomneo is a global initiative to simplify modern app development — a developer toolkit for fast, modular, multi-platform apps, engineered for AI collaboration and scalable architecture.

Vision

Empower developers to build fast, effortlessly

We're building a world where any developer — from solo builder to enterprise team — can scaffold a production-ready fullstack app in minutes, not days. No framework lock-in. No boilerplate rituals. Just code that works.

The problem we're solving

Four frustrations we refuse to accept

🧱

Complex frameworks

Steep learning curves slow teams down before a single line of product code is written. We believe setup should take minutes, not days.

🤖

Unreliable AI code

LLMs hallucinate APIs for large, undocumented codebases. Every Bloomneo package ships llms.txt and AGENTS.md so AI agents know exactly what's real.

🔒

Rigid architectures

Monoliths that can't grow. We designed FBCA so every feature is isolated, discoverable, and independently deployable from day one.

🔁

Slow workflows

Rebuilding the same auth, user management, and database patterns on every project. Bloom templates let you start with production patterns already in place.

Core values

What we stand for

  • Simplicity Clarity over complexity. Tools that are easy to learn, easy to use, and easy to leave if you outgrow them.
  • Autonomy Loosely coupled modules. Use AppKit without UIKit. Use UIKit without Bloom. Take what you need.
  • Modularity Features are isolated by design. A feature added today doesn't break something you built last month.
  • AI Innovation We treat AI coding agents as first-class consumers of our packages. Every API decision considers: will an LLM get this right?
  • Open Ecosystem MIT licensed. No vendor lock-in. No hidden tiers for core features. The framework is the product — not the pricing page.
  • Reliability Auto-scaling from memory to Redis to cloud without code changes. Features you ship in dev work the same in production.
Design principles

How we make decisions

Convention over configuration

If the right choice is obvious, it shouldn't require configuration. FBCA auto-discovers features. AppKit auto-scales storage. Bloom auto-mounts routes.

One pattern, everywhere

xxxClass.get() is the only way to instantiate any AppKit module. No constructors, no factories, no special cases — ever.

Start simple, scale naturally

In-memory cache → Redis. Local storage → S3. Console email → Resend. Every module upgrades by setting an env var. No code changes.

Minimal code, maximum intent

We'd rather ship one method that does the right thing than four methods that cover every edge case. Clear, purposeful APIs with smart defaults.

What we avoid

Things we refuse to build

  • BloatNo heavy dependencies. No features that don't earn their weight.
  • Tight couplingModules must stay independent. A change in one module never breaks another.
  • AI overpromisesWe don't claim AI generates perfect apps. We make AI agents reliably correct on our APIs.
  • Paywalled core featuresEverything fundamental is open source. If you need it to build, you get it for free.
  • Breaking changes without noticeSemver is a contract. We version, we document, we migrate.
Our commitment

We commit to providing intuitive tools and clear documentation, maintaining a free open-source ecosystem, listening to the community to evolve Bloomneo, and celebrating every contribution that advances the vision.

As the Bloomneo community, we pledge to build a platform that simplifies development, fosters collaboration, and inspires innovation — tools that empower developers and redefine how fullstack apps are built.
